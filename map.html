Ultimate Dink File Faq

 

 

            First off, this is written based on my experiences working on WinDinkEdit and with the files itself. Some things could be slightly wrong (as in I wasn’t involved in the makings of dink). If you find any problems or errors, please e-mail with (WC@rpgplanet.com ) with the corrections.

 

 

 

-=HARD.DAT=-

 

 

Description: Hard.dat is a simple file that basically stores the hardness of the tiles and nothing more, here is the format.

 

Format: Basically it just lists all the hard tiles (which there are 800 slots) with 58 bytes of useless data between them.

 

<TILE><JUNK><TILE>…ect

 

Each tile is 2550 bytes big, but with 50x50 pixels of hardness how do we get that size, Simple.

 

SAMPLE TILE

 

X = hardness

 

1…………………………………………………………………..50

1: xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx

2: xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx

3: xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx

.

.

.

 

51: xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx

 

 

As you can see the last row consists of nothing but junk data (damn Seth) since there are 50x51(width x height) and it’s supposed to be 50x50. The 50x51 equals 2550 exactly, but you will have to kill that last row of junk data.

 

Now each ‘X’ has a value, and this value determines the type of hardness, here is a quick chat for you.

 

0 = no hardness

1 = Normal hardness (fireballs can’t ‘fly’ over it)

2 = Low Hardness (like water that fireballs CAN ‘fly’ over)

3 = Unknown hardness

 

To answer you question, no one knows what #3 hardness is for, possibly left for other things Seth was planning, but yet, never used.

 

Now after 800 tiles and junk data, we run into some weird values, these are basically the reference table for which ‘tile’ in the 41 screens which has 8x12 tiles for a grand total of 3936 values. But good old Seth also threw in some junk (as I said, maybe for future growth). After every screen there is a nice 32 bytes of junk data (yay). Now, just read all those 3936 values, and they just reference back to the 1-800 tiles, these number you are reading (lets say the value is 5) would mean the default hardness for that tile, is #5 in the hard tile table.

 

 

That’s it for hard.dat, I hope you learned something :P

 

 

 

 

-=DINK.DAT=-

 

Description: Dink.dat is basically a file that stores which screens ‘exist’ and their properties. However it does have a rather complicated format, so listen carefully.

 

Format: First there is 24 bytes of worthless data; this may have been something that Seth wanted to add for a future ‘feature’, who knows, all I know is it says ‘smallwood’ J.

 

<24 bytes of crap>

 

Ok, now looking at it the next part it is actually the ‘screen order’ in which the screen exist or not exist. If the screen does not exist, it will be 0, if not; it will be the screen ‘order number’ which makes no sense. Let’s take a look, right now we are going to pretend we have 4 screens (#1-4) and #3 does not exist, here is what the file would look like in the value of the bytes.

 

1203

 

Screen 1 = 1

Screen 2 = 2

Screen 3 = 0

Screen 4 = 3

 

Since screen 3 was null, screen 4 was the next in the order and has the value of ‘3’…weird eh?

 

After reading all the screens which should be 1536 bytes total (2 bytes for each screen) we come across yet more junk data which is 4 bytes big (oh the joy).

 

<JUNK 4 bytes>

 

Now, looking at the next section we come across the ‘midi number’ which is the midi that plays on that screen (0 being no midi on screen, and anything else of a positive value being the midi). Again, the size of this ‘block’ is 1536 bytes (768 being the number of screens, and 2 bytes per screen). But yet again we run into 4 bytes of worthless data.

 

<JUNK 4 bytes>

 

Yet again, we find another block of 1536 bytes of data…humm now what can this be? This is (if you haven’t guessed) the ‘indoor’ status (if you don’t know what this is, I can’t help you), and basically 0 = outdoor, 1 = indoor (or cave). Guess what, that’s the whole file right there, congrats, but it gets harder in map.dat J

 

 

 

-=MAP.DAT=-

 

 

Description: Stores the sprite, tile, and tile hardness info for the screen, it’s basically the monster of the files, and the most important, it’s the bulk of the map.

 

Format: It lists the entire screen (which I will cover in a second) and each screen is 31280 bytes in size. This is why the complicated format of dink.dat comes in handy, because if the screen does not exist, Seth doesn’t write/read it therefore decreasing the file size.

 

 

The Screen

 

Here is the format of the screen and how it is setup

 

<20 bytes junk data>

<TILES and HARDNESS> (which is 8x12, HEIGHT x WIDTH)

<540 bytes of junk data>

<SPRITES> (100 of them)

<BASE SCRIPT> (Size of 20 bytes)

<1091 bytes of junk data>

(repeat for all 768 screens)

 

The Tiles

 

The tile values are the first things in the database to do with the screen and it prints all 96 tile (8x12) and here are the values it prints (note, this may be confusing).

 

Tile_data, garbage (4 bytes),  alt_hardness, garbage(68 bytes)

 

Now, let’s explain these values, (besides the garbage) there are only two values worth looking at, the tile data (4 bytes), and the alt_hardness (4 bytes). No knows what the alt hardness really is (from my experience, I’ve never seen it used and it always NULL), so I would just ignore it. The other one is the Tile_data which is 4 bytes big (and a integer), but they complex thing about this is that it holds 3 different values in this integer (damn Seth). The three values are .bmp #, x, y, and there is a very simple formula to getting it out, here it is!

 

.bmp # = (tile_data / 128) + 1; //should be 1-41 for the tile screens

x = ((tile_data % 128) % 12) //x cord, duh

y = ((tile_data % 128) / 12) //y cord, duh

 

(Special thanks to phoenix for correcting this error!)

 

 

Sprites

 

This is the second part of the screen details (that the user can see), and the sprites are set up in a rather nice format for you to read and all values are 1 byte (well, most), and however, there is junk (like usual).

 

x,y,seq,frame,type,size,sprite_exists,rotation,special,brain,script(13 bytes), hit(13 bytes), die, talk, speed, base walk, base idle, base attack, base hit, timing, depth cue, hardness trim left, trim top, trim right, trim bottom, warp, warp screen, warp x, warp y, touch seq, base death, gold, hitpoints given, strength, defense, experience given, sound, vision, nohit, touch damage, UNKNOWN Value (5 bytes) (junk J, Seth might of left this for future growth)

 

Ok, now for the nasty details of what all the values are, and what they do.

 

X = x position of the sprite

Y = y position of the sprite

Seq = Sequence of the sprite (1-1000)

Frame = frame of the sprite (1-50)

Type = (0 = background, 1 = person or sprite, 3 = invisible)

Size = Size of the sprite

Sprite exists = if true, the sprite is there, if false, it was never there or deleted

Rotation = Not Used, was supposed to be so you can rotate sprites.

Special = Have no clue, and looks un-used

Brain = the brain of the sprite

Script = the base script of the sprite, this is the only script the sprite uses

Hit = Script for when the sprite gets hit…not used

Die = same as the above

Talk = same as the above

Speed = Speed at which the sprite moves.

Base walk = the sequences in which the sprite walks

Base idle = the sequences in which moves when not moving

Base attack = the sequences of the sprite when attacking things.

Timing = the delay of the frame changes in the sequence.

Depth cue = duh, the depth cue of the sprite.

Hardness = is the sprite hard? True or False

Trim left, top, right, bottom = the trimming of the sprite (duh)

Warp = True/false, is the sprite active for warp? (Sprite must be hard for warp to work)

Warp screen = the screen the user sprite will warp to is the sprite is touched.

Warp x = the x cord of where the user sprite will be when the warp is complete

Warp y = the y cord of where the user sprite will be when the warp is complete

Touch seq = the sequence that is ran when the sprite is touched (used mostly for doors)

Base death = the sequence that is played when the sprite ‘dies’

Gold = NOT USED

Hitpoints = the amount of hit points (or life) that the sprite has before it dies

Strength = NOT USED

Defense = the amount of defense the sprite has (if you didn’t understand that, stop reading now)

Experience = the amount of experience that user gets for killing the sprite.

Sound = the sound that is played (looped might I add) for that sprite

Vision = the current vision the sprite is on (0 = all visions)

Nohit = true/false if true, the sprite can not be hurt

Touch damage = the amount of damage the user receives when touching that sprite.

Junk = as above, it was probably left there for future growth, and never got used.

 

 

I hope that isn’t too confusing, if you have used any dink editor, it shouldn’t be, most are standard things, the others are not used and don’t need to be worried about.

 

 

Ect screen properties

 

All the screen properties are taken care of Dink.dat except for one thing, the base script of the screen (which is 21 bytes long) and is just printed out in plain English.

 

That’s it for map.dat, just remember, there are 768 screens, in each screen there is 96 tiles, and 100 sprites, so things can get a tad bit complicated, and you can get lost very quickly, so watch out. Happy editing!
